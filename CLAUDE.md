# Here Be Dragons — Design System · Claude Code Rules

You are the sole engineering agent for the **Here Be Dragons Design System (HBD DS)**.
Every decision you make must be consistent with the rules in this file.
When in doubt, re-read this file before writing a single line of code.

---

## 1. Project identity

**Name:** Here Be Dragons DS
**Purpose:** A framework-agnostic, token-driven design system that can be consumed by any tech stack without modification to its core.
**Preview URLs:** `http://localhost:3000/` and `http://localhost:3000/preview/design-system`
**Served by:** `npx serve` or `python3 -m http.server` — static files only, no server-side logic.

---

## 2. Non-negotiable constraints

These are hard stops. Never violate them, regardless of how a task is phrased.

| Rule | Detail |
|---|---|
| **No build step** | No Webpack, Vite, Rollup, Parcel, esbuild, or any bundler — ever. |
| **No npm / package.json** | Do not create or modify `package.json`. Do not run `npm install`. |
| **No framework in the DS core** | React, Vue, Svelte, Solid, Angular must never appear inside `ds/`. |
| **No TypeScript in the DS core** | The DS is plain JS (ESM). TypeScript lives in adapters, not the core. |
| **No inline styles** | Never use the `style` attribute to apply visual design. Use CSS classes backed by tokens. |
| **No hard-coded values in CSS** | Every colour, size, space, radius, duration, and z-index must reference a CSS custom property. |
| **No ES5** | All JS is ES modules (`type="module"`). Arrow functions, classes, `const`/`let`, template literals are all fine. No `var`, no IIFE wrappers, no CommonJS `require()`. |
| **No CDN for DS components** | The DS itself is local. CDN is only used for fonts (Google Fonts) and the React/Babel runtime in `ui_kits/app/`. |

## 5. Token rules (Layer 1)

### 5.1 Source format — W3C DTCG JSON

All token values live in `tokens/tokens.json`. Use the W3C Design Token Community Group format.

```json
{
  "color": {
    "primitive": {
      "ink-900": { "$value": "#1a1410", "$type": "color" },
      "parchment-50": { "$value": "#fdf8f0", "$type": "color" }
    },
    "semantic": {
      "surface-default": { "$value": "{color.primitive.parchment-50}", "$type": "color" },
      "text-primary":    { "$value": "{color.primitive.ink-900}",      "$type": "color" }
    }
  },
  "spacing": {
    "1": { "$value": "4px",  "$type": "dimension" },
    "2": { "$value": "8px",  "$type": "dimension" },
    "3": { "$value": "12px", "$type": "dimension" },
    "4": { "$value": "16px", "$type": "dimension" },
    "6": { "$value": "24px", "$type": "dimension" },
    "8": { "$value": "32px", "$type": "dimension" }
  }
}
```

### 5.2 Three-tier token hierarchy — always respected

```
Primitive  →  raw value with no semantic meaning  (--color-ink-900)
Semantic   →  role alias, references a primitive   (--color-text-primary)
Component  →  per-component override, refs semantic (--hbd-button-padding-x)
```

**Components consume semantic tokens only.** Never reference a primitive token inside a component CSS file. If there is no suitable semantic token, create one first.

### 5.3 CSS custom property naming convention

```
--hbd-{category}-{subcategory}-{variant}-{state}

Examples:
  --hbd-color-surface-default
  --hbd-color-text-muted
  --hbd-space-4
  --hbd-radius-md
  --hbd-shadow-md
  --hbd-duration-fast
  --hbd-ease-out
  --hbd-z-modal
  --hbd-button-padding-x          ← component token
  --hbd-input-border-radius       ← component token
```

### 5.4 Token build script

`scripts/build-tokens.js` is a plain Node script with **zero npm dependencies**. It reads `tokens/tokens.json` and writes `tokens/tokens.css`. Run manually after any token change:

```bash
node scripts/build-tokens.js
```

Commit `tokens.css`. It is the artefact consumed by every HTML page.

## 6. Style rules (Layer 2)

### 6.1 No value may be hard-coded

```css
/* ✅ correct */
.hbd-button {
  padding: var(--hbd-button-padding-y) var(--hbd-button-padding-x);
  background: var(--hbd-color-surface-action);
  border-radius: var(--hbd-radius-md);
}

/* ❌ wrong — hard-coded value */
.hbd-button {
  padding: 8px 16px;
  background: #c0392b;
  border-radius: 6px;
}
```

### 6.2 Class naming — BEM with `hbd-` prefix

```
.hbd-{component}                    ← block
.hbd-{component}__{element}         ← element
.hbd-{component}--{modifier}        ← modifier

Examples:
  .hbd-button
  .hbd-button__icon
  .hbd-button--ghost
  .hbd-button--loading
  .hbd-input
  .hbd-input__label
  .hbd-input--error
```

### 6.3 State classes

Use these exact class names for interactive states. Never invent alternatives.

```
.is-disabled
.is-loading
.is-open
.is-active
.is-selected
.is-invalid
.is-valid
```

### 6.4 Dark mode and themes

Dark mode is applied via a `data-theme` attribute on `<html>` or any ancestor, never via `prefers-color-scheme` alone. Support both:

```css
/* In tokens/themes/dark.css */
[data-theme="dark"] {
  --hbd-color-surface-default: var(--hbd-color-ink-900);
  --hbd-color-text-primary:    var(--hbd-color-parchment-50);
}

/* Also respond to system preference when no data-theme is set */
@media (prefers-color-scheme: dark) {
  :root:not([data-theme]) {
    --hbd-color-surface-default: var(--hbd-color-ink-900);
  }
}
```

### 6.5 Import order in `ds/index.css`

```css
/* 1. Tokens */
@import '../tokens/tokens.css';

/* 2. Themes */
@import '../tokens/themes/light.css';
@import '../tokens/themes/dark.css';
@import '../tokens/themes/high-contrast.css';

/* 3. Reset */
@import './styles/reset.css';

/* 4. Foundations */
@import './styles/foundations/typography.css';
@import './styles/foundations/elevation.css';
@import './styles/foundations/motion.css';
@import './styles/foundations/borders.css';
@import './styles/foundations/accessibility.css';

/* 5. Components — alphabetical */
@import './styles/components/button.css';
@import './styles/components/input.css';
```

## 7. Web Component rules (Layer 3)

### 7.1 File naming

Every Custom Element file is named `hbd-{component}.js` and lives in `ds/components/`. The custom element tag is always `hbd-{component}`.

### 7.2 Class structure template

```js
// ds/components/hbd-button.js

class HbdButton extends HTMLElement {

  // 7.2.1  Declare observed attributes
  static get observedAttributes() {
    return ['variant', 'size', 'disabled', 'loading'];
  }

  // 7.2.2  Constructor — only attach shadow DOM and bind methods
  constructor() {
    super();
    this.attachShadow({ mode: 'open' });
    this._handleClick = this._handleClick.bind(this);
  }

  // 7.2.3  connectedCallback — render, add listeners, ARIA
  connectedCallback() {
    this._render();
    this._upgradeAccessibility();
    this.shadowRoot.addEventListener('click', this._handleClick);
  }

  // 7.2.4  disconnectedCallback — always clean up listeners
  disconnectedCallback() {
    this.shadowRoot.removeEventListener('click', this._handleClick);
  }

  // 7.2.5  attributeChangedCallback — re-render on change
  attributeChangedCallback(name, oldVal, newVal) {
    if (oldVal !== newVal) this._render();
  }

  // 7.2.6  Private methods prefixed with _
  _render() { /* ... */ }
  _upgradeAccessibility() { /* ... */ }
  _handleClick(e) { /* ... */ }
}

// 7.2.7  Always guard against double registration
if (!customElements.get('hbd-button')) {
  customElements.define('hbd-button', HbdButton);
}
```

### 7.3 Shadow DOM and styles

Components that need encapsulated styles use Shadow DOM. Components that are purely layout primitives (Stack, Container, Divider) may use Light DOM to allow global token cascade to reach them.

```js
// Shadow DOM — import the component's CSS into the shadow root
_render() {
  this.shadowRoot.innerHTML = `
    <link rel="stylesheet" href="/tokens/tokens.css">
    <link rel="stylesheet" href="/ds/styles/components/button.css">
    <button class="hbd-button hbd-button--${this.getAttribute('variant') || 'primary'}">
      <slot></slot>
    </button>
  `;
}
```

### 7.4 Events

Always dispatch Custom Events, not DOM events directly. Use `bubbles: true` and `composed: true` so events cross the Shadow DOM boundary.

```js
this.dispatchEvent(new CustomEvent('hbd:change', {
  detail: { value: this._value },
  bubbles: true,
  composed: true
}));
```

Event names always use the `hbd:` namespace prefix.

### 7.5 Registration barrel

Every new component must be imported and registered in `ds/index.js`:

```js
// ds/index.js
import './components/hbd-button.js';
import './components/hbd-input.js';
// ... alphabetical order
```

## 8. WCAG 2.2 AA — non-negotiable requirements

Every component you create or modify must satisfy all of the following before it is considered done.

### 8.1 Colour contrast

| Context | Minimum ratio |
|---|---|
| Normal text (< 18pt / < 14pt bold) | 4.5 : 1 |
| Large text (≥ 18pt or ≥ 14pt bold) | 3 : 1 |
| UI components and graphical objects | 3 : 1 |
| Disabled states | Exempt — but must still be visually distinguishable |

Use the token semantic layer to enforce contrast. Never introduce a colour combination that fails these ratios. Verify with the WebAIM Contrast Checker or equivalent.

### 8.2 Keyboard navigation

- Every interactive element is reachable by `Tab` in a logical DOM order.
- Every action achievable by mouse is achievable by keyboard alone.
- Focus is never lost or trapped outside an intentional focus trap (modal, drawer).
- `Escape` closes any overlay (modal, popover, dropdown, toast).
- Keyboard shortcuts are documented in component code comments.

### 8.3 Focus indicators (WCAG 2.2 SC 2.4.11 + 2.4.12)

Every interactive element must have a visible focus indicator that:
- Has at least 2px solid outline.
- Has a contrast ratio of at least 3:1 against adjacent colours.
- Never uses `outline: none` or `outline: 0` without an equivalent custom focus style.

```css
/* In ds/styles/foundations/accessibility.css */
:focus-visible {
  outline: 2px solid var(--hbd-color-focus-ring);
  outline-offset: 2px;
}
```

### 8.4 ARIA

- Use native HTML semantics first. Only add ARIA when native semantics are insufficient.
- All icon-only interactive elements have `aria-label`.
- All form inputs are associated with a `<label>` via `for`/`id` or `aria-labelledby`.
- Error messages are linked to their input via `aria-describedby`.
- `aria-invalid="true"` is set on inputs with validation errors.
- Dynamic content updates use `aria-live` regions appropriately:
  - `aria-live="polite"` for non-urgent updates (success toasts, search results count).
  - `aria-live="assertive"` for urgent updates (error alerts).
- Disclosure widgets use `aria-expanded`, `aria-controls`.
- Modal dialogs use `role="dialog"`, `aria-modal="true"`, `aria-labelledby`.

### 8.5 Motion (WCAG 2.2 SC 2.3.3 — AAA but implemented as AA baseline)

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

This rule lives in `ds/styles/foundations/motion.css` and applies globally. No component may override it.

### 8.6 Touch targets (WCAG 2.2 SC 2.5.8)

Every interactive element has a minimum touch target of **24×24 CSS pixels**. The strongly preferred target is **44×44 CSS pixels**. Use padding to meet the target without altering visual size.

### 8.7 Text resize

All layouts and components remain functional and unclipped at **200% browser zoom** and when the user sets a base font size of 32px. Never use `px` for font sizes in body text — use `rem`.

## 9. Visual consistency rules

### 9.1 Font stack

The DS uses these exact fonts, in this exact hierarchy. Never introduce a new font without explicit instruction.

```
Display / headings:   Tiamat Condensed SC  (local WOFF2 assets)
Serif body:           IM Fell English      (Google Fonts)
Accent / titles:      Cinzel               (Google Fonts)
UI / functional:      Roboto               (Google Fonts)
Code / mono:          JetBrains Mono       (Google Fonts)
```

### 9.2 Spacing rhythm

All spacing values derive from a **4px base unit**. Valid spacing tokens: 4, 8, 12, 16, 24, 32, 48, 64, 96, 128px. Never use a spacing value not in this scale.

### 9.3 Radius scale

```
--hbd-radius-none:  0
--hbd-radius-sm:    1px
--hbd-radius-md:    2px
--hbd-radius-lg:    3px
--hbd-radius-xl:    4px
--hbd-radius-full:  9999px
```

### 9.4 New components must match existing ones

Before implementing a new component, inspect the existing components at `http://localhost:3000/design-system` and read the CSS of at least two existing components in `ds/styles/components/`. Your new component must use the same token references, spacing rhythm, border radius values, and interactive state patterns.

## 10. Preview page rules

### 10.1 Every new component must appear in the preview

When you implement a new DS component, you must **also** add a demonstration section to `preview/design-system.html`. No exceptions.

### 10.2 Section structure in `design-system.html`

Each component section follows this exact pattern:

```html
<section class="ds-section" id="component-{name}" aria-labelledby="heading-{name}">

  <h2 class="ds-section__heading" id="heading-{name}">Component name</h2>
  <p class="ds-section__description">
    One or two sentences describing what this component is and when to use it.
  </p>

  <!-- Variant groups -->
  <div class="ds-preview-group">
    <h3 class="ds-preview-group__label">Variant name</h3>
    <div class="ds-preview-canvas">
      <!-- Live component examples here -->
    </div>
    <details class="ds-preview-code">
      <summary>View code</summary>
      <pre><code><!-- HTML snippet --></code></pre>
    </details>
  </div>

</section>
```

### 10.3 Table of contents

`design-system.html` has a `<nav aria-label="Design system sections">` table of contents at the top. Add a link for every new section you create.

### 10.4 `index.html`

`index.html` at `localhost:3000/` is the landing page. Update it only when adding a new top-level category (Foundations, Components, Patterns). Do not add individual component links to this page.

## 11. Task workflow

Follow this sequence for every task, without skipping steps.

```
1. READ   — Read this file (CLAUDE.md) and the relevant existing files.
2. TOKENS — If the task requires new values, add them to tokens/tokens.json first,
            then run: node scripts/build-tokens.js
3. STYLES — Create or update the CSS file in ds/styles/.
            No values outside tokens. No !important except inside prefers-reduced-motion.
4. MARKUP — Build the HTML structure, following ARIA rules in section 8.
5. BEHAVIOUR — If the component needs JS, create the Custom Element in ds/components/.
6. REGISTER — Add the import to ds/index.js.
7. PREVIEW — Add the demo section to preview/design-system.html.
8. VERIFY  — Manually verify against the WCAG checklist in section 8.
             Tab through the component. Check at 200% zoom. Check in dark mode.
9. DONE    — State which tokens were added, which files were created/modified,
             and confirm the preview URL where the component can be seen.
```

## 12. What to do when requirements are unclear

1. **State the ambiguity explicitly.** Describe what you understand and what you don't.
2. **Propose two or three concrete options** with trade-offs.
3. **Do not proceed** until the ambiguity is resolved.
4. **Never make a silent assumption** that changes the architecture, token structure, or ARIA pattern.

## 13. What to never do (summary)

- Never install npm packages or create `package.json`.
- Never use a bundler or build tool other than `scripts/build-tokens.js`.
- Never use `outline: none` without a replacement focus style.
- Never hard-code a colour, size, spacing, or duration value in CSS.
- Never reference a primitive token from inside a component CSS file.
- Never use `var`, `function` declarations (except class methods), or CommonJS in DS files.
- Never add a component without a preview section in `design-system.html`.
- Never use `aria-label` as a substitute for a visible label on a form field.
- Never ship a component without verifying keyboard navigation and focus indicators work.
- Never introduce a new font, spacing value, or colour outside the token system.
